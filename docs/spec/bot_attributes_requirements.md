## 1. 背景と目的

### 1.1 背景

現行の yuuka は、全Botインスタンスが「パーソナル秘書」として固定のフル機能セット
（ToDo・家計・リマインド・ブラウザ操作・マクロ等の全 FunctionModule）を持つ。
Botインスタンスの複数管理（§5.1 Bot単位の定義）は既に可能だが、
**インスタンスごとに機能セットを変える概念が存在しない**。

### 1.2 目的

Botインスタンスに **「属性（Bot属性）」** を導入し、同一基盤上で性格の異なる
LLM統合Botを作成できるようにする。

- パーソナル秘書（現行機能、デフォルト）
- **MCPアシスタント（汎用モード）**: MCP接続・ペルソナ・メモリのみで構成される簡易Bot。
  Discordサーバー（ギルド）に常駐し、**利用メンバー**（自然言語で気軽に追加できる、§4.3.3）が
  接続済みMCPサーバーのツール群を通じてサーバー運用を便利にできる（第一弾の追加対象）
- 将来: 属性の組み合わせによる任意構成のLLM統合Bot

### 1.3 第一弾のゴール

「MCP接続 + ペルソナ + メモリ」のみを持つ簡易Bot属性を1つ追加し、
属性を保持・判定する基盤（DB列・Function組み立て・管理UI）を整備する。
秘書機能の細分化（家計だけのBot等）は本書のスコープ外（将来課題）。

## 2. 用語定義

| 用語 | 定義 |
|------|------|
| **Bot属性 (bot attribute)** | Botインスタンスが持つ機能ケーパビリティの集合。Botインスタンス単位で保持する |
| **ケーパビリティ (capability)** | 属性を構成する最小単位。FunctionModule 群・システムプロンプト断片・管理UI設定タブの束に対応する |
| **プリセット (preset)** | ケーパビリティの定義済み組み合わせ。Bot作成時に選択する |
| **MCPアシスタント（汎用モード）** | 第一弾で追加するプリセット。`mcp` + `persona` + `memory` で構成。ユーザー向け表示名は「汎用モード」を想定（§10-5） |
| **メモリ** | 記憶機構の総称。(a) 会話コンテキストと (b) コンテキストノートの2層【2026-06-13 確定】。汎用モードでは (a) はギルド単位・直近30件、(b) は個人ノート（bot×ユーザー）と共有ノート（bot×ギルド）の2種を持つ（§4.6） |

## 3. 属性モデルのコンセプト

### 3.1 方式の比較

| 方式 | 内容 | 利点 | 欠点 |
|------|------|------|------|
| A. enum型 | `bots.bot_type = "secretary" \| "mcp_assistant"` | 実装が最小 | 「いろんなBotを作れる」将来像で型が爆発する |
| B. ケーパビリティ集合 + プリセット | `bots.capabilities = JSON配列`。作成時はプリセットから選択 | 将来の組み合わせ自由化に直結。判定は `has(cap)` で統一 | 初期実装がやや大きい |

**採用方針（推奨）: 方式B**。
ただし第一弾ではユーザーに見せる選択肢をプリセット2種（秘書 / MCPアシスタント）に限定し、
ケーパビリティの自由編集UIは作らない（将来課題）。

### 3.2 ケーパビリティ一覧（第一弾）

| capability | 内容 | 対応する既存実装 |
|------------|------|------------------|
| `core` | LLM対話・リッチ返信・会話履歴記録。**全Bot必須**（暗黙付与、JSONには書かない） | gemini.ts, message_logs |
| `persona` | ペルソナ適用・推奨ペルソナ | personas, personaRepo |
| `memory` | コンテキストノート（個人/共有）+ 会話ログ検索・要約（汎用モードではギルド単位） | context_notes, noteFunctions, conversationFunctions |
| `mcp` | MCPサーバー接続・動的Function | mcp_servers, mcpDynamic, mcpClient |
| `secretary` | 現行秘書機能一式（ToDo・スケジュール・リマインド・家計・ブラウザ・マクロ・クリップボード・連絡先・朝報/日報・Webhook・PWマネージャ・グラフ） | 上記以外の全 FunctionModule |

> `secretary` は第一弾では分割しない単一ケーパビリティとする。
> 将来 `finance` / `browser` 等へ細分化する余地を残す（その際も JSON 配列なら互換を保てる）。

### 3.3 プリセット定義（第一弾）

| プリセット | capabilities | 想定用途 |
|------------|-------------|----------|
| `secretary`（パーソナル秘書） | `["persona","memory","mcp","secretary"]` | 現行と完全同一。既存Botのデフォルト |
| `mcp_assistant`（MCPアシスタント） | `["persona","memory","mcp"]` | ギルド常駐のサーバー便利Bot |

> プリセットの**ユーザー向け表示名は管理ページから変更可能**とする【2026-06-13 決定】。
> 既定値は「パーソナル秘書」「汎用モード」。内部ID（`secretary` / `mcp_assistant`）は固定とし、
> 表示名は system_settings に保持する（変更権限は Admin を仮置き。§10-5）

## 4. 機能要件

### 4.1 Bot属性の管理

- Bot作成時（管理UI: Bot管理）にプリセットを選択する。未選択時は `secretary`
- 属性の変更は **Bot作成者（owner）のみ** が管理UIから実行できる
- 属性変更は即時反映（次のメッセージ処理から新しいFunctionセットを使用）
- 属性は監査ログ（audit_logs）に記録する（action: `bot.capabilities_change`）

### 4.2 Function組み立てへの反映

- `processMessage`（gemini.ts）で Bot の capabilities を解決し、
  **保持しないケーパビリティの FunctionModule は declarations / dispatch の両方から除外**する
  （LLMに宣言自体を見せない。description での抑制は不可）
- システムプロンプト（`buildSystemInstruction`）も capabilities に応じて組み立てる。
  MCPアシスタントには秘書機能のルール文（承認フロー・記憶ツール使い分け等のうち秘書系）を含めない
- cron系サービス（リマインド・朝報・日報等）は `secretary` を持つBotのみを対象とする

### 4.3 MCPアシスタント（第一弾プリセット）の要件

#### 4.3.1 構成

- **MCP接続**: Botに紐付けたMCPサーバーのツールのみを動的Functionとして提供（§4.5）
- **ペルソナ**: Bot単位ペルソナを適用（§4.4）
- **メモリ**: 会話コンテキスト + コンテキストノート（§4.6）
- 上記以外のFunctionは一切持たない

#### 4.3.2 応答スコープ【2026-06-13 決定】

- 許可ギルド（bot_guilds）内の「メンション / Botへの返信」に応答する
- **DM には応答しない。ただし Bot作成者（owner）からの DM のみ応答する**（動作確認・管理用途）。
  owner との DM は専用コンテキストで扱い、ギルドのコンテキストとは分離する（§4.6.1）

#### 4.3.3 利用者（利用メンバー制）と実行APIキー【2026-06-13 決定】

- 利用できるのは、許可ギルド内の **利用メンバー（bot_members）** とする。
  Webアカウント登録・Bot共有（§5.2）は不要（DiscordユーザーIDのみで管理する）
- **メンバーの追加・削除**:
  - Bot作成者（owner）が管理UIから追加・削除できる（owner 自身は常に暗黙メンバー）
  - 利用メンバーは、Botへの自然言語の依頼（例: 「@xx を追加して」）で**新規メンバーを追加**できる
  - 削除は本人（「私を外して」）と owner のみ可能（メンバーによる他者削除は不可。§10-10 仮置き）
  - メンバー管理は専用 Function 群（addBotMember / removeBotMember / listBotMembers）として
    汎用モードの core に含める
- メンバー外のギルドメンバーがメンションした場合は、LLMを呼ばず定型応答で利用案内
  （「メンバーに追加してもらうと利用できます」等）を返す。
  **メンバー外の発話はログ・ノートに一切記録しない**（未登録は無視。2026-06-13 決定）
- LLM呼び出しは **Bot専用のGemini APIキー** で実行する【2026-06-13 決定】:
  - 汎用モードのBot作成時に**設定必須**とする。未設定・無効化されたキーのBotは応答せず、
    管理UIに警告を表示する
  - owner の個人キー（users テーブル）とは**別管理**とし、bots テーブルへ暗号化保存する。
    発話ユーザーが登録済みユーザーであっても本人キーは使用しない
- Bot専用キー保護のため、以下の利用量制御を必須とする（詳細は §6）:
  - **ギルド許可リスト**: owner が管理UIで許可したギルドでのみ応答する。
    未許可ギルドに招待された場合は応答も記録もしない（誰でもBotをギルドへ追加できるため必須の防衛線）
  - **レート制限**: ユーザー単位・ギルド単位の利用回数上限。超過時はLLMを呼ばず定型応答を返す

### 4.4 ペルソナ（Bot単位ペルソナ）

- MCPアシスタントでは、**Bot作成者がBotインスタンスに対して1つのペルソナを直接設定**する
  （`bots.persona_id`）。ギルドの全利用者に同じ人格で応答するため
- 設定可能なペルソナは owner 所有のもの、または `is_public = 1` のもの
- 未設定時は既存のデフォルトペルソナ（gemini.ts の DEFAULT_PERSONA）を使用
- **ペルソナの設定・変更は管理ページからのみ**行える【2026-06-13 決定】。
  会話（自然言語）からのペルソナ変更 Function は提供せず、利用者ごとの上書きも不可
- 秘書プリセットは現行どおり**ユーザー単位**（users.active_persona_id）を維持し、
  `bots.persona_id` は参照しない（優先順位の混在を避ける）

### 4.5 MCP接続（Bot単位の紐付け）

- Bot作成者は、**自身が登録したMCPサーバー**（mcp_servers, user_id = owner）を
  Botインスタンスへ紐付け/解除できる（複数可）
- MCPアシスタントが利用できるツールは「紐付けられたサーバー + システムレベル(user_id NULL)サーバー」のみ。
  発話ユーザー個人のMCPサーバーは**参照しない**（秘書プリセットとの差分）
- `requires_confirmation` は既存仕様（§4.4.3）を踏襲。デフォルト有効
- tools_cache の更新・Function名正規化（`mcp{serverId}_{toolName}`）は既存実装（mcpDynamic.ts）を流用
- **注意**: これは「MCPサーバーは登録者本人のみ利用可」とする現行仕様 §4.4.3 の改訂を伴う（§9 参照）

### 4.6 メモリ【2026-06-13 決定: 会話コンテキスト + コンテキストノートの2層】

#### 4.6.1 会話コンテキスト（ギルド単位・直近30件）

- 汎用モードの会話コンテキストは **bot_id × ギルド単位** でスコープする。
  ギルド内の会話は利用メンバー間で共有される1つの流れとして扱う
  （チャンネル単位の分離は行わない。必要になれば Phase 2 で検討）
- Redis キー: `context:{botId}:{guildId}` を新設。**直近キャッシュは30件**
  （秘書の15件から拡張。複数人の発話が混ざるギルドでは文脈量を倍にする）。
  秘書の `context:{userId}` とは完全に分離し、秘書との会話はギルドBotに漏れない
- 会話履歴の記録は既存 message_logs を流用し、**guild_id 列を追加**して発話ギルドを記録する
  （NULL = DM・秘書利用）。発話者の user_id も従来どおり記録する
- 会話ログ検索・要約（searchConversationLogs 等）は **bot_id × ギルド単位** で動作する。
  検索対象はそのギルドでの会話のみ（他ギルド・DM・秘書の会話は対象外）
- owner との DM（§4.3.2）は `context:{botId}:dm:{ownerId}` で分離する（直近15件）

#### 4.6.2 コンテキストノート（個人 + 共有の両方を持つ）

- **個人ノート（bot_id × ユーザー単位）**: 利用メンバーごとの記憶。
  本人の依頼・会話からのみ書き込まれ、他メンバーのプロンプトには注入しない
- **共有ノート（bot_id × ギルド単位）**: ギルド共有の知識ベース
  （サーバーのルール・用語・運用手順等）。利用メンバー全員が参照・編集できる（§10-11 仮置き）
  - 補足: §10-2 の選択肢では「Bot共通」としていたが、1つのBotは複数ギルドに参加できるため、
    **ギルド単位**にスコープしてギルド間の情報漏えいを防ぐ
- メモリ Function 群は、個人ノート用と共有ノート用を LLM が明確に使い分けられる宣言
  （名前・description）で提供する（命名は実装時に決定。例: 個人= myNote系 / 共有= guildNote系）
- プロンプトへの注入順: ペルソナ → 共有ノート → 発話者の個人ノート

### 4.7 管理UI

- Bot管理ページ: 作成フォームにプリセット選択を追加。Bot一覧に属性バッジを表示
- システム設定（Admin）: プリセット表示名の変更（§3.3）
- MCPアシスタント設定タブ（owner のみ）:
  - **Bot専用 Gemini APIキーの設定（必須項目）**
  - ペルソナ選択（自身のペルソナ + 公開ペルソナ）※変更はここからのみ（§4.4）
  - MCPサーバー紐付け管理（自身の登録済みサーバー一覧から選択、紐付け状態表示）
  - 応答許可ギルドの管理（bot_guilds）
  - 利用メンバーの管理（bot_members: 追加・削除・一覧）
  - 共有ノート（ギルドノート）の閲覧・編集
- 属性に含まれない機能のタブ/設定はそのBotの文脈では非表示

## 5. データモデル変更（概略）

migrations.ts（唯一の定義元）への追加。日時は既存スタイル（localtime テキスト）。

```
bots（既存テーブルへ列追加）
  + capabilities : TEXT NOT NULL DEFAULT '["persona","memory","mcp","secretary"]'
                   (JSON配列。既存行はマイグレーションでデフォルト値=秘書相当)
  + persona_id   : INTEGER NULL  (MCPアシスタント用Bot単位ペルソナ。FK: personas.id)
  + gemini_api_key_encrypted / iv / auth_tag : TEXT NULL
                   (Bot専用Gemini APIキー。システム鍵で暗号化。汎用モードでは設定必須)

bot_mcp_links（新設: BotとMCPサーバーの紐付け）
  id            : INTEGER PK AUTOINCREMENT
  bot_id        : TEXT NOT NULL      (bots.id)
  mcp_server_id : INTEGER NOT NULL   (mcp_servers.id, owner所有のもののみ)
  created_at    : TEXT
  UNIQUE(bot_id, mcp_server_id)

bot_context_notes（新設: 個人ノート。context_notes は秘書用に現状維持）
  bot_id     : TEXT NOT NULL
  user_id    : TEXT NOT NULL   (発話ユーザー。アカウント未登録のDiscordユーザーIDも可)
  content    : TEXT NOT NULL DEFAULT ''  (上限は context_notes と同じ 10,000 文字)
  updated_at : TEXT
  PRIMARY KEY (bot_id, user_id)

bot_guild_notes（新設: ギルド共有ノート）
  bot_id     : TEXT NOT NULL
  guild_id   : TEXT NOT NULL   (DiscordギルドID)
  content    : TEXT NOT NULL DEFAULT ''  (上限 10,000 文字)
  updated_at : TEXT
  PRIMARY KEY (bot_id, guild_id)

bot_guilds（新設: MCPアシスタントの応答許可ギルド）
  bot_id     : TEXT NOT NULL   (bots.id)
  guild_id   : TEXT NOT NULL   (DiscordギルドID)
  created_at : TEXT
  PRIMARY KEY (bot_id, guild_id)

bot_members（新設: 汎用モードの利用メンバー）
  bot_id     : TEXT NOT NULL   (bots.id)
  guild_id   : TEXT NOT NULL   (DiscordギルドID)
  user_id    : TEXT NOT NULL   (DiscordユーザーID。アカウント未登録可)
  added_by   : TEXT NOT NULL   (追加操作を行ったユーザーID)
  created_at : TEXT
  PRIMARY KEY (bot_id, guild_id, user_id)
```

- レート制限カウンタは Redis（例: `mcp_rate:{botId}:{userId}` / `mcp_rate_guild:{botId}:{guildId}`、
  TTL付きカウンタ）で保持し、SQLiteには持たない。上限値の既定はシステム設定とし、
  Bot単位の上書き列（bots へ追加）は Phase 2 で検討

- message_logs: **guild_id 列（TEXT NULL）を追加**（NULL = DM・秘書利用。bot_id 列は既存）。
  ギルド単位検索用のインデックスを追加する
- Redis: キー体系の追加のみ（§4.6.1。汎用モードのギルドコンテキストは直近30件）
- mcp_servers: スキーマ変更なし（紐付けは bot_mcp_links 側で表現）

## 6. セキュリティ・権限要件

| 操作 | User | Bot作成者(owner) | Admin |
|------|:----:|:----:|:-----:|
| Bot作成時のプリセット選択 | —(作成者のみ) | ✅ | ✅ |
| Bot属性の変更 | ❌ | ✅ | ✅ |
| BotへのMCPサーバー紐付け | ❌ | ✅(自身の登録サーバーのみ) | ✅ |
| Bot単位ペルソナの設定 | ❌ | ✅ | ✅ |
| 応答許可ギルドの管理（bot_guilds） | ❌ | ✅ | ✅ |
| MCPアシスタントの利用（対話） | ✅(利用メンバーのみ。Webアカウント不要) | ✅ | ✅ |
| 利用メンバーの追加（自然言語 / 管理UI） | ✅(メンバーのみ、自然言語) | ✅(両方) | ✅ |
| 利用メンバーの削除 | ✅(自分自身のみ) | ✅(任意) | ✅ |
| 共有ノートの参照・編集（会話経由） | ✅(メンバーのみ) | ✅ | ✅ |
| 自身の個人ノート（bot_context_notes）の閲覧・削除 | ✅(Web UIは要アカウント) | ✅ | ✅ |

- MCPツール実行は発話ユーザーを actor として audit_logs に記録（既存方針を踏襲）
- Botに紐付いたMCPサーバーの認証情報が共有ユーザーに露出しないこと
  （ツール名・description のみ公開。endpoint_url・認証情報は owner 以外に表示しない）
- `requires_confirmation = 1` のツールは現行の2段階確認フロー（architecture_v2.md §1）を必須とする
- データ分離原則（architecture_v2.md §0-1: user_id 必須）に対し、
  bot_context_notes 等の **bot_id × user_id 複合スコープ**を正式な例外パターンとして規範に追記する

**Bot専用キー実行に伴う利用量制御（必須）**

- **ギルド許可リスト**: bot_guilds に登録されたギルドでのみメッセージを処理する。
  未許可ギルドからのメッセージは黙殺（ログのみ）とし、LLM・MCPは呼ばない
- **レート制限**: ユーザー単位（既定: 5回/分・100回/日）・ギルド単位（既定: 1,000回/日）。
  既定値はシステム設定で変更可能（具体値は §10-8 仮置き）。
  超過時はLLMを呼ばず定型応答（「しばらく時間をおいてください」等）を返す
- **コスト可視化**: owner が管理UIでBot別の利用回数（日次）を確認できること
- 利用メンバーの発話・ツール実行は message_logs / audit_logs に DiscordユーザーIDで記録される
  （Webアカウント未登録のメンバーも同様）。
  **メンバー外の発話は記録しない**（応答は §4.3.3 の定型案内のみ。未登録は無視）

## 7. 非機能要件・後方互換

- **既存Botへの無影響**: マイグレーションで既存 bots 行に秘書相当の capabilities を付与し、
  動作・プロンプト・Functionセットが現行と完全に一致すること
- capabilities の解決はメッセージ処理のホットパスに乗るため、Bot設定はインメモリキャッシュ
  （属性変更時に無効化）とし、1メッセージあたりの追加DBアクセスを増やさない
- 新規 npm 依存の追加は不可（architecture_v2.md §0-6）。既存実装の流用で完結させる
- ドキュメント: 本要件確定後、discordbot_spec.md / architecture_v2.md へ反映する（§9）
