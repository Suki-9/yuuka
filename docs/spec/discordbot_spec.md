## 1. 概要

本仕様書は **DiscordBot** を中心とした統合管理ソフトウェアの設計・仕様を定義する。  
本ソフトウェアはDiscord上のBotとしての機能に留まらず、スケジュール管理・家計管理・ブラウザ自動操作など複数のツール群を統合的に管理するプラットフォームとして設計される。

### スコープ

| 対象 | 説明 |
|------|------|
| Discord Bot | ユーザーとのLLMベース対話 |
| 管理UI / API | Bot・ユーザー設定の管理インターフェース |
| 外部連携 | Google Calendar, Gemini API, ブラウザ操作ツール |
| データ基盤 | ユーザーデータの保存・分離・セキュリティ |

### スコープ外

- Discord以外のチャットプラットフォームへの対応（将来課題）
- Botのホスティングインフラ構築手順（別途インフラ仕様書で定義）
- スマートホームデバイス連携（Google Home / Matter 等）— 将来課題として余白を確保（§4.4 MCP拡張経由での接続を想定）

## 3. Botの機能要件

### 3.0 リッチ返信共通仕様

#### 3.0.1 概要

Bot の返信は **プレーンテキスト** と **リッチ形式** の2種類を使い分ける。リッチ形式は全機能を横断して適用可能な共通機構として実装し、各機能モジュールがこの仕様に従って返信を構築する。

LLMが返信内容を生成する際、応答の性質（データ量・構造化の有無・視覚的な有用性）を判断してリッチ形式の使用可否を自律的に決定する。

#### 3.0.2 形式: Discord Embed

Discord の [Embed オブジェクト](https://discord.com/developers/docs/resources/message#embed-object) を使用した色付きカード形式。

**使用場面の目安**

| 返信の種類 | Embed 使用 | 例 |
|-----------|:---------:|-----|
| 単純な一問一答 | ❌ | 「明日の予定は？」→ 箇条書きテキスト |
| データの一覧・サマリ | ✅ | タスク一覧、家計サマリ、連絡先詳細 |
| 定期配信（朝報・日報・週報） | ✅ | 天気・ニュース・週次レポート |
| 確認プロンプト（承認待ち） | ✅ | マクロ実行確認、支払い消込確認 |
| エラー・警告通知 | ✅ | Webhook受信エラー、API失敗通知 |
| 会話テキストの補足情報 | ✅ | 検索結果の要約、文字起こし結果 |

**Embedの構成要素と使用方針**

| フィールド | 用途 |
|-----------|------|
| `title` | 返信の見出し（機能名・イベント名など） |
| `description` | 本文・要約テキスト |
| `fields` | 構造化データ（名前・値のペア、最大25件） |
| `color` | 機能カテゴリ・状態を色で表現（下表参照） |
| `thumbnail` | 天気アイコン・連絡先アバターなど小画像 |
| `footer` | タイムスタンプ・データソース表記 |
| `image` | グラフ・チャート画像（§3.0.3 参照） |

**カラーコード規約**

| 用途 | 色 | HEX |
|------|----|-----|
| 通常情報 | ブルー | `#5865F2` |
| 成功・完了 | グリーン | `#57F287` |
| 警告・注意 | イエロー | `#FEE75C` |
| エラー・失敗 | レッド | `#ED4245` |
| 天気・朝報 | スカイブルー | `#00B0F4` |
| 家計・支払い | ゴールド | `#F1C40F` |
| タスク・スケジュール | パープル | `#9B59B6` |

#### 3.0.3 形式: グラフ・チャート画像

数値データを視覚化したグラフ画像を生成し、Embed の `image` フィールドまたはファイル添付として送信する。

**使用場面**

| 返信の種類 | グラフ種別 |
|-----------|----------|
| 家計サマリ（カテゴリ別支出） | 円グラフ / 棒グラフ |
| 家計サマリ（月次推移） | 折れ線グラフ |
| タスク完了率 | 棒グラフ / ドーナツグラフ |
| 天気予報（気温推移） | 折れ線グラフ |
| 予算消化率 | 横棒グラフ（プログレスバー風） |

**実装方針**

- グラフ生成ライブラリは **Chart.js**（Node.js環境で `chartjs-node-canvas` を使用）を採用する
- 生成した画像はPNGバイナリとしてDiscord APIにマルチパート送信する
- グラフのテーマ（フォント・背景色）はDiscordのダークモードに合わせたデザインとする

#### 3.0.4 対話フローへの組み込み

§3.1.2 の対話フローにリッチ返信の判断ステップを追加する。

```
1. メッセージ受信
2. コンテキスト取得（ペルソナ + コンテキストノート + 会話履歴）
3. Gemini API へ送信
4. Function Call → 各ツールモジュールが実行・結果を返す
5. LLM が返信内容を生成
6. [Rich Reply Router]
   - データ量・構造が Embed に適するか判断
   - グラフ生成が有用か判断
   - プレーンテキスト / Embed / Embed+グラフ のいずれかを選択
7. Discord へ送信
```

#### 3.0.5 ユーザー設定

- ユーザーはリッチ返信の有効/無効をユーザー設定で切り替えられる（デフォルト: 有効）
- 無効にした場合はすべての返信がプレーンテキストになる
- 機能単位（天気のみ無効など）の細かな制御は将来課題とする

### 3.1 LLM対話エンジン

#### 3.1.1 使用モデル

- **LLM:** Google Gemini API
- モデルバージョンは設定ファイルで切り替え可能とする

#### 3.1.2 対話フロー

1. DiscordメッセージをBot Applicationが受信
2. ユーザーIDに紐づくペルソナ・コンテキストノート・会話履歴をデータストアから取得
3. Gemini APIへプロンプトを送信（ペルソナ + コンテキストノート + 会話履歴 + ユーザーメッセージ）
4. Geminiの応答がFunction Callを含む場合 → Function Call Routerへディスパッチ
5. LLMが返信内容を生成
6. Rich Reply Routerが返信形式（プレーンテキスト / Embed / Embed+グラフ）を判断（§3.0 参照）
7. Discordへ返信

#### 3.1.3 Function Call設計方針

- Gemini の Function Calling 機能を用いてツール呼び出しを行う
- 各ツールモジュールはFunction Callのスキーマ（name / description / parameters）を定義し、ルーターに登録する
- LLMは原則としてユーザーの意図を解釈した上で適切なFunctionを選択する

#### 3.1.4 会話コンテキスト

- 会話履歴はユーザーID単位で **Redis** に直近15件をキャッシュし、Gemini APIへの送信に使用する
- 全ての会話履歴（送受信メッセージ）は **SQLiteにも永続保存**する（件数・期間の制限なし）
- Redisキャッシュが失われた場合はSQLiteから直近15件を復元して再構築する
- コンテキストのキー: `context:{discord_user_id}`（Redisのリスト型、末尾が最新）
- Redis TTL: セッション有効期限（7日間）に連動してリセット

**返信チェーンの解決**

- メッセージが **返信（Reply）形式** の場合、返信元メッセージも取得してコンテキストに含める
- 返信チェーンは **再帰的に複数段階まで遡る**（ルートメッセージまで辿る）
- チェーン解決で取得したメッセージは、通常の15件キャッシュとは別枠でコンテキストの先頭に追加する
- チェーン遡りの上限深度は実装時に設定値として定義する（無限ループ防止）

### 3.2 スケジュール管理・タスク優先度整理・タグ管理

#### 3.2.1 機能一覧

| 機能 | 説明 |
|------|------|
| ToDo追加 | タスクをユーザーのToDoリストに追加する |
| ToDo一覧表示 | 未完了・完了済みタスクを一覧・タグ別・グループ別に表示する |
| ToDo完了・削除 | タスクのステータスを変更・削除する |
| Googleカレンダー登録 | 指定した日時・件名でカレンダーにイベントを作成する |
| Googleカレンダー照会 | 指定期間のイベントを取得し表示する |
| Googleカレンダー削除 | 指定イベントを削除する |
| タスク優先度整理 | 未完了ToDoをLLMが分析し、優先順位の提案を行う |
| タグ自動付与 | タスク登録・更新時にLLMがタグを自動付与しグルーピングする |
| サブタスク | タスクに1階層のサブタスクをぶら下げ、分解した手順を管理する（§3.2.6） |
| 進捗管理 | 進捗率(0-100%)を更新し、進捗メモを履歴として保存する（§3.2.6） |
| ガントチャート | 開始日・期限を持つタスクをWeb管理画面にガントで表示する（§3.2.6） |

#### 3.2.2 Google Calendar連携

- OAuth 2.0 によりユーザー毎に認証・認可を取得する
- リフレッシュトークンはユーザーデータストアに暗号化して保存する
- スコープ: `https://www.googleapis.com/auth/calendar`

#### 3.2.3 タスク優先度整理

- ユーザーが「タスクを整理して」などと依頼した際に、未完了ToDoの全件をLLMに渡して優先順位を分析・提案する
- 分析の考慮要素: 期限（`due_date`）の近さ、タイトル・説明文から読み取れる重要度、Googleカレンダーの予定との兼ね合い、タグ・グループ
- LLMは提案のみを行い、順序の確定・並び替えはユーザーの承認後にのみ実行する
- 優先度は `TodoItem` に `priority` フィールド（`high` / `medium` / `low`）を持たせて保存する

#### 3.2.4 タグ管理・グルーピング

**タグ付与の仕組み**

- タスクが追加・更新されるたびに、LLMがタイトル・説明文・コンテキストノートをもとにタグを自動付与する
- タグは複数付与可能（例: `["業務", "開発", "締め切り間近"]`）
- タグ候補はユーザーの既存タスクから学習した語彙を優先し、新規語彙は必要に応じて追加する
- タグ付与はバックグラウンドで非同期に行い、ユーザーへの応答をブロックしない

**グルーピングの仕組み**

- 同一タグを持つタスクを自動的に同グループとして扱い、一覧表示時にまとめて表示できる
- 代表的なグループ例: `日常`, `業務`, `買い物`, `支払い`, `学習` など（LLMが命名）
- グループ名はそのタスク群の中からLLMが最も代表的なタグを選択して命名する

**タグの管理操作**

- ユーザーはタグ一覧の閲覧・特定タグのタスク一覧表示を自然言語で依頼できる（例: 「業務タスクを見せて」）
- タグの明示的な削除は行わず、そのタグを持つタスクが0件になれば自動的に非表示になる

#### 3.2.5 ToDoデータモデル（概略）

```
TodoItem {
  id                : UUID
  user_id           : DiscordUserId
  title             : string
  description       : string (optional)
  due_date          : datetime (optional, ガントのバー終端)
  start_date        : datetime (optional, v12: ガントのバー始端)
  priority          : "high" | "medium" | "low" (optional, LLM自動付与)
  tags              : JSON    (string[], LLM自動付与, 例: ["業務", "開発"])
  status            : "open" | "done"
  progress          : int 0-100 (v12: 手動進捗。子を持つ親は算出値を表示)
  parent_id         : UUID (optional, v12: 自己参照。サブタスクの親。1階層のみ)
  linked_payment_id : UUID (optional, 支払い予定との紐付け)
  created_at        : datetime
  updated_at        : datetime
}

TaskProgressLog {            // v12: 進捗報告の時系列ログ
  id         : UUID
  user_id    : DiscordUserId
  bot_id     : BotId
  todo_id    : UUID
  progress   : int 0-100     // この時点の進捗
  note       : string (optional, 例: "設計フェーズ完了")
  created_at : datetime
}
```

#### 3.2.6 進捗管理・サブタスク・ガント（v12）

**サブタスク**

- タスクは `parent_id` による自己参照で **1階層** のサブタスクを持つ（サブタスクは孫を持てず、指定された場合は親へ平坦化される）
- 親タスクの削除時はサブタスク・進捗ログも連鎖削除される
- LLMツール: `addSubtask`（親IDを指定して追加）

**進捗管理**

- 進捗は **算出ルール** で表現する:
  - サブタスクを持つ親タスク: `完了サブタスク数 / 全サブタスク数 × 100`（自動算出・手動更新不可）
  - サブタスクを持たない葉タスク: 手動 `progress`（`status=done` は100扱い。100更新で `done`・100未満で `open` へ同期）
- 「〜まで終わった」等の進捗報告で `progress` を更新し、`TaskProgressLog` に履歴を1件追記する（経緯を後から追える）
- LLMツール: `updateTaskProgress`（進捗率＋メモ）、`getTaskDetail`（サブタスク・算出進捗・進捗履歴）

**ガントチャート（Web管理画面）**

- タスクビューの「一覧 / ガント」トグルで切替。`chart.js`（self-host）の水平フローティングバーで描画
- バーは `start_date` → `due_date`。進捗分の塗り・今日ラインを重ねる
- **開始日・期限がともに未設定のタスクはガントに載せず**「いつかやる」セクションへ振り分ける
  - 期限のみ → 単日マイルストーン / 開始日のみ → 今日まで（進行中扱い）

### 3.3 リマインド機能

#### 3.3.1 機能一覧

| 機能 | 説明 |
|------|------|
| 時刻指定リマインド | 指定日時にDiscordのDM/チャンネルへメッセージを送信する |
| タスク起因リマインド | ToDoの期限が近づいた際に自動通知する |
| スケジュール起因リマインド | Googleカレンダーのイベント前に自動通知する |
| 繰り返しリマインド | 日次・週次など定期的なリマインドを設定できる |

#### 3.3.2 リマインド実行基盤

- スケジューラは **cron** を使用する
- cronジョブが定期的にSQLiteのReminderテーブルを検索し、`trigger_at <= now()` かつ `status = "pending"` のエントリを処理する
- リマインドの送信先チャンネル / DM はユーザー設定に従う
- 通知前時間のデフォルト値はユーザーが設定可能とする（例: イベント10分前）
- 繰り返しリマインドの `repeat_rule` はcron式（例: `0 9 * * 1` = 毎週月曜9時）で保存する

#### 3.3.3 リマインドデータモデル（概略）

```
Reminder {
  id           : UUID
  user_id      : DiscordUserId
  message      : string
  trigger_at   : datetime
  repeat_rule  : string (optional, cron形式)
  target       : { type: "dm" | "channel", id: string }
  status       : "pending" | "sent" | "cancelled"
}
```

### 3.4 家計管理・支払い予定

#### 3.4.1 機能一覧

| 機能 | 説明 |
|------|------|
| 収支手動入力 | 金額・カテゴリ・メモを指定して収支を記録する |
| レシート画像入力 | レシート画像をDiscordに送信すると自動でOCR・記帳する |
| 収支一覧・集計 | 期間・カテゴリ別の収支を表示する |
| 予算管理 | カテゴリごとに月次予算を設定し、消化率を通知する |
| 支払い予定の登録 | 予定金額・概要・期日を登録し、家計に反映する |
| 支払い予定からToDo生成 | 支払い予定をToDoタスクとして追加する |
| 支払い予定からリマインド設定 | 支払い期日前にリマインドを自動設定する |
| 支払い予定の消込 | 実際の支払い記録と予定を紐付けて管理する |

#### 3.4.2 レシート自動記帳フロー

1. ユーザーがレシート画像をDiscordに添付してメッセージ送信
2. BotがDiscord Attachment URLから画像を取得
3. Gemini Vision APIに画像を送信し、品目・金額・日付を構造化データとして抽出
4. 抽出結果をユーザーに確認プレビューとして返信
5. ユーザーが承認 → 家計データストアへ保存
6. 承認時、対応する支払い予定（`PlannedPayment`）が存在する場合は消込候補として提示する

#### 3.4.3 支払い予定の仕様

**登録**

- 「〇〇の支払いが△日に□□円ある」などの自然言語からLLMが構造化して登録する
- 繰り返し支払い（家賃・サブスクなど）はcron式で周期を保存し自動的に次回予定を生成する

**ToDo・リマインド連携**

- 支払い予定登録時に「ToDoとして追加する？」「リマインドを設定する？」の確認プロンプトを表示する
- どちらも選択した場合、`TodoItem.linked_payment_id` と `Reminder` を同時に生成する
- ToDoのタグはLLMが `["支払い", "<カテゴリ名>"]` を自動付与する

**消込フロー**

1. 実際の支払いを `Expense` として記録する（手動入力またはレシートOCR）
2. `PlannedPayment` と金額・カテゴリ・日付が近い `Expense` を自動照合してペアを提示する
3. ユーザーが承認 → `PlannedPayment.status` を `"settled"` に更新し、`settled_expense_id` を記録する
4. 消込済みの予定に紐付いたToDoが存在する場合、自動的に完了（`done`）に変更する

#### 3.4.4 収支データモデル（概略）

```
Expense {
  id         : UUID
  user_id    : DiscordUserId
  type       : "income" | "expense"
  amount     : integer (円単位)
  category   : string
  memo       : string (optional)
  date       : date
  source     : "manual" | "receipt_ocr"
  created_at : datetime
}

PlannedPayment {
  id                 : UUID
  user_id            : DiscordUserId
  title              : string          (概要)
  amount             : integer         (予定金額、円単位)
  category           : string
  due_date           : date            (支払い期日)
  repeat_rule        : string (optional, cron式)
  status             : "pending" | "settled" | "cancelled"
  settled_expense_id : UUID (optional, 消込したExpenseのID)
  linked_todo_id     : UUID (optional, 紐付きToDoのID)
  linked_reminder_id : UUID (optional, 紐付きReminderのID)
  created_at         : datetime
  updated_at         : datetime
}
```

### 3.6 操作記憶（マクロ）機能

#### 3.6.1 概要

Botに対して操作を実行させるか手順を説明した後、「覚えておいてね」などの記憶指示を伝えることで、その操作手順をマクロとして保存し、以降短いフレーズで再呼び出しできる機能。

#### 3.6.2 登録フロー

**実行ベース登録（Botに操作させた場合）**

1. ユーザーがBotに操作を実行させる（例: ブラウザ操作、Function Call等）
2. 実行後に「覚えておいてね」「これを記憶して」などの記憶指示を送信
3. LLMが直前の操作履歴から実行内容を抽出・要約し、マクロ候補として提示する
4. ユーザーが呼び出し名（例: 「朝の確認」）を確定 → SQLiteに保存

**説明ベース登録（手順を説明した場合）**

1. ユーザーが手順を自然言語で説明する（例: 「〇〇サイトを開いて△△ボタンを押して...」）
2. 説明末尾に記憶指示を添える
3. LLMが手順をFunction Call列として構造化し、マクロ候補として提示する
4. ユーザーが確認・呼び出し名を確定 → 保存

#### 3.6.3 呼び出し

- 登録した呼び出し名またはそれに近い表現をDiscordで送信すると、LLMがマクロとマッチングして実行を提案する
- 実行前にユーザーへ確認メッセージを送り、承認後に実行する
- マクロ一覧の表示・削除・編集も可能とする

#### 3.6.4 マクロデータモデル（概略）

```
Macro {
  id          : UUID
  user_id     : DiscordUserId
  name        : string              (呼び出し名)
  description : string              (LLMが生成した手順の要約)
  steps       : JSON                (Function Call列またはブラウザ操作ステップ列)
  created_at  : datetime
  updated_at  : datetime
}
```

### 3.7 メモリ / コンテキストノート

#### 3.7.1 概要

ペルソナ（Bot全体の人格設定）とは独立して、ユーザーが「覚えておいてほしい事柄」を永続メモとして保存する機能。会話のたびにシステムプロンプトへ注入することで、LLMが毎回参照できる。

#### 3.7.2 ペルソナとの違い

| | ペルソナ | コンテキストノート |
|---|---------|-----------------|
| 用途 | Botの人格・話し方・キャラクター設定 | ユーザー固有の考慮事項・背景知識 |

| 上限 | 20,000文字 | 10,000文字（ペルソナと合算でコンテキストを圧迫しないよう制限） |

#### 3.7.3 登録・管理

- 「〇〇を覚えておいて」「メモしておいて」などの自然言語でノートに追記される
- LLMが既存ノートと重複・矛盾する情報を検出した場合は確認を求める
- 管理UIまたはコマンドからノート全体の参照・編集・削除が可能
- ノートはシステムプロンプトの末尾（ペルソナの後）に挿入される

#### 3.7.4 コンテキストノートデータモデル（概略）

```
ContextNote {
  id          : UUID
  user_id     : DiscordUserId
  content     : string    (全文、10,000文字以内)
  updated_at  : datetime
}
```

### 3.8 日報・週報の自動生成

#### 3.8.1 概要

#### 3.8.2 生成内容

| セクション | データソース |
|------------|-------------|

ReportConfig {

  target        : { type: "dm" | "channel", id: string }
  updated_at    : datetime
}
```

### 3.10 クリップボード / 一時メモ

#### 3.10.1 概要

コンテキストノート（§3.7）が「長期的な背景知識」であるのに対し、クリップボードは「今日・今だけ覚えておいてほしい揮発的なメモ」を扱う機能。TTLを持ち、期限を過ぎると自動削除される。

#### 3.10.2 コンテキストノートとの違い

| | コンテキストノート | クリップボード |
|---|-----------------|--------------|
| 用途 | 長期的な背景知識・ユーザー属性 | 短期的な一時メモ・作業中の情報 |
| 例 | 「私はエンジニア」「締め切りは金曜」 | 「今日の会議メモ」「あとで調べるURL」 |

| 件数制限 | 1件（全文管理） | 複数件（エントリ単位） |

#### 3.10.3 登録・管理

- 「これをメモしておいて」「クリップしておいて」などの自然言語で登録する
- 登録時にTTLを指定できる（例: 「今週中だけ」「1時間後に消して」）
- TTLはcronジョブで定期チェックし、期限切れエントリを自動削除する
- 「メモを見せて」で一覧表示、「〇〇のメモを消して」で手動削除できる

#### 3.10.4 クリップボードデータモデル（概略）

}
```

### 3.12 会話ログ要約

#### 3.12.1 概要

SQLiteに保存された全会話履歴（§7）を対象に、特定トピックの過去のやり取りを時系列順に取り出してLLMが要約する機能（`summarizeConversationTopic`）。「先週の〇〇の議論をまとめて」などの自然言語クエリに対応する。時系列の文脈が本質的に重要なため、シナプスの連想想起では代替できない機能として維持する。

> **シナプス認知アーキテクチャとの整合（重複の解消）:** LLMが明示的に呼んだ時だけ動く受動的な会話全文検索 `searchConversationLogs`（旧 §3.12 のキーワード検索）は、過去の知見を能動的・連想的に思い出す **シナプス L2 連想想起**（architecture_v2.md §13）が置き換えるため**廃止**した。本節は時系列要約のみを規定する。

#### 3.12.2 機能一覧

| 機能 | 説明 |
|------|------|
| トピック要約 | キーワード・期間で特定した過去のやり取りを時系列順に取り出し、LLMが要約して返す |
| 件数の絞り込み | 該当が多い場合は新しい順の上位N件に絞る（Nはデフォルト10件） |

#### 3.12.3 実装方針

- 要約対象の特定にSQLiteの全文検索（FTS5拡張）と期間条件を用いる（キーワードはあれば併用、無ければ期間のみ）
- 期間・キーワードの解釈はLLMがFunction Callの引数として構造化して渡す
- 取得ログは時系列（古い順）で返し、要約自体はLLMが行う
- プライバシー配慮のため、対象は本人（汎用モードはそのギルド）の会話のみ（他ユーザー・他ギルドには非表示）
- 過去の知見の能動的・連想的な想起は本機能ではなくシナプス L2 連想想起が担う（architecture_v2.md §13）

### 3.13 外部Webhook受信

#### 3.13.1 概要

GitHub・Zapier・決済サービス等の外部サービスからWebhookを受け取り、Botがその内容を処理・通知する機能。MCPサーバー拡張（§4.4）が「BotからMCPへの送信」であるのに対し、Webhookは「外部からBotへの受信」として補完的に機能する。

#### 3.13.2 仕組み

- Bot Application ServerがHTTPエンドポイントを公開し、Webhookを受信する
- ユーザーは管理UIでWebhookエンドポイントを作成し、発行されたURLを外部サービスに登録する
- 受信したWebhookペイロードをLLMが解釈し、ユーザーのDiscord DM / チャンネルへ通知する
- 通知内容のフォーマットはユーザーがテンプレートで設定できる

#### 3.13.3 機能一覧

| 機能 | 説明 |
|------|------|
| Webhookエンドポイント作成 | ユーザー固有のWebhook URLを発行する |
| ペイロード解釈・通知 | 受信したJSONをLLMが解釈してDiscordへ通知する |
| フィルタリング | 条件に合致するペイロードのみ通知する（例: issueのクローズ時のみ） |
| ToDoへの変換 | 受信内容をToDoタスクとして自動登録できる |
| リマインドへの変換 | 受信内容からリマインドを自動生成できる |

#### 3.13.4 セキュリティ

- Webhookエンドポイントは推測困難なランダムトークンを含むURLとする
- **HMAC 署名検証用シークレットを必須**とする（作成時に 16 文字以上を要求し、署名なし受信は拒否する）。シークレットはシステム鍵で暗号化保存する
- 受信ペイロードは処理後にDBへ監査ログとして保存する

#### 3.13.5 Webhookデータモデル（概略）

```
WebhookEndpoint {
  id               : UUID
  user_id          : DiscordUserId
  name             : string
  token            : string          (URLトークン、CSPRNG生成)
  secret           : string (必須, HMAC検証用シークレット≥16文字、暗号化保存)
  notify_target    : { type: "dm" | "channel", id: string }
  template         : string (optional, 通知テンプレート)
  enabled          : boolean
  created_at       : datetime
}
```

### 3.14 音声メモの文字起こし

#### 3.14.1 概要

Discordに送信された音声ファイル・ボイスメッセージをGeminiのマルチモーダル機能で文字起こしし、テキストとして保存・活用する機能。

#### 3.14.2 機能一覧

| 機能 | 説明 |
|------|------|
| 文字起こし | 音声ファイル（ogg, mp3, wav, m4a等）をテキスト化する |
| 要約 | 文字起こし結果をLLMが要約して返す |
| ToDoへの変換 | 「〇〇しておいて」を含む音声メモを自動でToDoに変換する |
| クリップボード保存 | 文字起こし結果をクリップボード（§3.10）に保存する |
| 会話履歴への記録 | 文字起こし結果をMessageLogに保存し、将来の要約（§3.12）・シナプス連想想起の対象とする |

#### 3.14.3 処理フロー

1. ユーザーが音声ファイルをDiscordに添付して送信
2. BotがDiscord Attachment URLから音声ファイルを取得
3. Gemini APIのマルチモーダル機能で文字起こしを実行
4. 文字起こし結果をユーザーにプレビュー表示
5. ToDoへの変換候補が検出された場合は確認プロンプトを表示
6. MessageLogおよび（任意で）ClipboardEntryに保存

#### 3.14.4 対応フォーマット

Gemini APIがサポートする音声フォーマットに準じる（ogg, mp3, wav, m4a, flac 等）。

### 4.1 ペルソナ管理

#### 4.1.1 仕様

- ペルソナはユーザー毎に独立して保持される（他ユーザーへの影響なし）
- ペルソナの実体は**システムプロンプト文字列**であり、Gemini API呼び出し時に先頭に挿入される
- デフォルトペルソナは最低限の設定（丁寧なアシスタント）とする

#### 4.1.2 ペルソナのカスタマイズ

- ユーザーは管理コマンドまたは管理UIからペルソナを自由に編集できる
- 文字数上限: **20,000文字**（Geminiのコンテキストウィンドウに対して余裕を持たせた設定）

#### 4.1.3 ペルソナの共有

- ユーザーは自分のペルソナを「公開」設定にすることができる
- 公開されたペルソナはペルソナマーケットプレイス（またはリスト）に表示される
- 他ユーザーは公開ペルソナを「インポート」して自分のペルソナとして利用できる
- インポート後は独立したコピーとなり、元ペルソナの変更は反映されない
- Bot作成者は公開されているペルソナをそのBotの **「推奨ペルソナ」** として設定できる（§5.2 参照）

#### 4.1.4 ペルソナデータモデル（概略）

```

  name        : string
  prompt      : string
  is_public   : boolean
  created_at  : datetime
  updated_at  : datetime
}
```

### 4.3 Bot設定・同期

#### 4.3.1 Botトークンの管理

- BotのDiscordトークンは **Botをホストするユーザー（Bot管理者）のみ**が設定・変更できる
- Bot管理者権限は環境変数または専用の管理者設定ファイルで定義する

#### 4.3.2 アイコン・名前の同期

- BotのアイコンとBot名は**Discord Bot側の設定と常に同期**される
- 同期のタイミング: 起動時・定期同期（**1時間ごと**）・手動同期コマンド実行時
- 同期方法: Discord API（`PATCH /applications/@me`）を使用

#### 4.3.3 Botの起動・手動停止（schema v9）

- オーナーは統合管理画面から Bot を**手動停止**できる。停止状態（`bots.stopped=1`）は**永続化**され、プロセス再起動後も自動起動の対象外となる
- 管理者による処分停止（`suspended`）とは**独立**したフラグとする。`stopped` はオーナー操作・`suspended` は管理者操作
- 統合管理 API は Bot の起動 / 停止 / 再起動、および会話履歴クリア（§7）を提供する

### 4.4 MCPサーバー拡張

#### 4.4.1 概要

MCP（Model Context Protocol）サーバーを登録することで、標準機能にない操作をFunction Callとして追加できる拡張機構。ユーザーが任意のMCPサーバーを接続し、BotがそのToolをLLMのFunction Callとして利用できるようになる。

将来的には Google Home / Matter などのスマートホームデバイスや音声アシスタントとの連携も、対応するMCPサーバーを登録することで実現できる設計とする。現時点でスマートホーム連携の実装仕様は定義しないが、以下の点を設計上の余白として確保する。

- デバイス操作系のToolは `requires_confirmation = true` をデフォルトとし、誤操作を防ぐ
- デバイス状態取得（照明ON/OFF、温度など）はユーザーの明示的な許可後にのみ可能とする
- スマートホーム連携用MCPサーバーは将来的にシステムレベル登録（Admin管理）での提供を想定する

#### 4.4.2 登録フロー

1. ユーザーが管理UIまたはコマンドでMCPサーバーのエンドポイント（URL）・認証情報を登録する
2. BotがMCPサーバーの `tools/list` エンドポイントを呼び出し、提供Toolの一覧を取得する
3. 取得したToolスキーマをGemini APIのFunction Call定義として動的に登録する
4. 以降、LLMがユーザーの意図に応じてMCP Toolを呼び出せるようになる

#### 4.4.3 セキュリティ要件

- MCPサーバーの認証情報（APIキー等）はパスワードマネージャと同じ暗号化方式（AES-256-GCM）で保存する
- 登録できるMCPサーバーは登録したユーザー本人のみが利用できる（他ユーザーへの共有不可）
- MCPサーバーの呼び出し前にユーザーへ確認を求めるか否かは、Toolごとに `requires_confirmation` フラグで制御する
- `Admin` ロールはインスタンス全体で利用可能なシステムレベルのMCPサーバーを登録できる

#### 4.4.4 MCPサーバーデータモデル（概略）

```
McpServer {
  id                   : UUID
  user_id              : DiscordUserId   (NULLの場合はシステムレベル登録)
  name                 : string          (表示名)
  endpoint_url         : string
  auth_credential      : string          (暗号化保存)
  tools_cache          : JSON            (tools/list の取得結果キャッシュ)
  tools_cache_updated  : datetime
  enabled              : boolean
  created_at           : datetime
}
```

## 5. Botの単位とユーザー管理

### 5.1 Bot単位の定義

- **BotはDiscord APIトークン単位で1インスタンスとする**
- Botインスタンスはアカウントを持つユーザー（Bot作成者）が自身のアカウントで作成する
- 各インスタンスに対して共有設定を行うことで、複数ユーザーが同一Botを利用できる
- 共有する場合でも、各ユーザーのデータは完全に分離される（§5.5 参照）

---

### 5.2 Bot共有設定

#### 5.2.1 概要

- Botを作成したユーザー（**Bot作成者**）は、他のユーザーをそのBotに招待・共有できる
- 共有はBot作成者のみが管理できる（追加・削除・公開設定の変更）
- 共有設定はBotインスタンス単位で管理され、ユーザー単位の設定（APIキー・ペルソナ等）には影響しない
- Bot作成者は公開されているペルソナをそのBotの **「推奨ペルソナ」** として1つ設定できる
  - 推奨ペルソナはマーケットプレイス上で当該Botへの参加ユーザーに案内・提示される
  - ユーザーが推奨ペルソナをインポートするかどうかは任意であり、強制適用はされない
  - 推奨ペルソナは `is_public = true` のペルソナのみ設定可能とする

#### 5.2.2 共有フロー

1. Bot作成者が管理UIまたは管理コマンドで対象ユーザーのDiscordユーザーIDを指定して招待
2. 招待されたユーザーにDiscord DMで通知が届く（推奨ペルソナが設定されている場合はその情報も通知）
3. 招待ユーザーが承認 → そのBotへのアクセスが有効化される
4. 推奨ペルソナが設定されている場合、承認時にインポートするか確認プロンプトを表示する
5. Bot作成者はいつでも特定ユーザーのアクセスを取り消せる

#### 5.2.3 共有設定データモデル（概略）

```
BotInstance {
  id                     : UUID
  owner_id               : DiscordUserId   (Bot作成者)
  discord_token          : string          (暗号化保存)
  recommended_persona_id : UUID (optional) (is_public = true のペルソナのみ指定可)
  created_at             : datetime
}

BotShare {
  id            : UUID
  bot_id        : UUID            (BotInstanceの識別子)
  owner_id      : DiscordUserId   (Bot作成者)
  shared_user_id: DiscordUserId   (招待ユーザー)
  status        : "pending" | "active" | "revoked"
  created_at    : datetime
}
```

### 5.4 ユーザー登録・認証

#### 5.4.1 登録フロー

1. ユーザーが初回コマンド（例: `/register`）を実行
2. BotがDiscordのユーザーIDを自動取得
3. ユーザーがパスワードを設定（DMで送信 または Web管理UI上で入力）
4. パスワードはbcrypt（コストファクター: 12）でハッシュ化してSQLiteに保存
5. 初回登録ユーザーのロールは `User` として登録
6. 登録完了通知をDiscord DMで送信

> **DM 所有確認チャレンジ（必須）**: Web から任意の Discord ID で登録される「なりすまし」を防ぐため、登録は仮登録（メモリ保持・有効期限あり）とし、当該 Discord ID へ **ワンタイムコードを DM 送信**する。ユーザーがそのコードを提示して検証が成功した場合のみ、実ユーザーをDBに作成する（実装: `services/pendingRegistration.ts`）。

#### 5.4.2 認証フロー

- Discord上のやり取りでは、DiscordユーザーID＋セッショントークンによる認証を行う
- 管理UI（Web）へのログインは DiscordユーザーID + パスワード で行う
- **タイミングオラクル対策**: ログイン時、ユーザーが存在しない場合でも固定時間の bcrypt 比較（ダミー）を実行し、応答時間差によるアカウント列挙を防ぐ
- セッショントークンの仕様:
  - 有効期限: **7日間**
  - 更新ポリシー: 有効期限内にアクセスがあった場合、有効期限を自動延長（スライディングウィンドウ）
  - 保存場所: Redis（キー: `session:{token_hash}`）
  - 失効: ログアウト時またはパスワード変更時に即時無効化

#### 5.4.3 パスワードポリシー

- 最低8文字以上
- 大文字・小文字・数字・記号のうち **2種類以上** を含むこと
- よく使われるパスワード（上位10,000件）との一致をチェックし拒否する
- ハッシュアルゴリズム: **bcrypt**（コストファクター: 12）

#### 5.4.4 アカウント管理・本人によるアカウント削除

- 管理ダッシュボードに**アカウント管理ページ**を設け、表示名・テーマ・パスワード変更を集約する（Bot 個別画面からは分離）
- ユーザーは**自身のアカウントを削除**できる。要件:
  - 削除前に**現在のパスワード検証を必須**とする
  - **最後の管理者は削除不可**（`Admin` ロールが 1 名以下のとき防止）。サービスのロックアウトを避ける
  - 削除時、本人が所有する独立 Bot（`system_default` を除く）を停止し、ユーザーデータを破棄する

---

### 5.5 データ分離モデル

- **データベース:** SQLite（ファイルパスは環境変数で設定）
- 全てのユーザーデータは **DiscordユーザーID（`discord_user_id`）をWHERE句の必須条件**として分離する
- SQLiteはRLSをサポートしないため、**アプリケーションレベルのスコープフィルタ**を全クエリに必ず適用する
  - 例: `SELECT * FROM todos WHERE discord_user_id = ? AND id = ?`
  - ユーザーIDなしのワイルドカードクエリはアプリケーション層で禁止する
- Botトークンを共有していても、あるユーザーが他ユーザーのデータにアクセスすることは**不可能**とする
- `Admin` ロールであっても他ユーザーの個人データへの直接アクセスは禁止とし、管理操作を行った場合は監査ログを記録する

## 6. パスワードマネージャ（ブラウザ操作用）

### 6.1 概要

ブラウザ自動操作においてログインが必要なWebサービスへアクセスするため、パスワードマネージャ機能を内蔵する。LLMはFunction Callを通じてパスワードマネージャにアクセスし、認証情報を取得してブラウザに入力する。

### 6.2 暗号化方式

#### 6.2.1 保存時暗号化（Encryption at Rest）

| 項目 | 仕様 |
|------|------|
| 暗号化アルゴリズム | AES-256-GCM |
| 暗号化鍵の導出 | **環境変数のシークレット（`YUUKA_ENCRYPTION_SECRET`）+ ユーザー固有のソルト** をArgon2idに通して導出 |
| ソルト | ユーザー登録時にCSPRNGで生成、SQLiteに保存 |
| シークレット管理 | `YUUKA_ENCRYPTION_SECRET` は環境変数で注入し、コードやDBには含めない |
| データの暗号化単位 | 認証情報エントリ（サービス名・URL・ユーザー名・パスワード）ごとに個別のIVで暗号化 |
| IV（初期化ベクタ） | エントリ作成・更新ごとにCSPRNGで生成し、暗号文と共に保存 |

> **シークレットスクランブル方式の設計根拠:** Bot Application Serverがブラウザ操作を担うため完全なゼロ知識は実現困難であるが、`YUUKA_ENCRYPTION_SECRET` を環境変数で管理することでDBが漏洩しても平文パスワードを取得できない構成とする。

> **`YUUKA_ENCRYPTION_SECRET` ローテーション手順:**
> 1. 新しい `YUUKA_ENCRYPTION_SECRET` を生成し、環境変数 `YUUKA_ENCRYPTION_SECRET_NEW` として追加設定する
> 2. Bot起動時に移行スクリプトを実行: 全暗号化エントリを旧キーで復号 → 新キーで再暗号化 → SQLiteに書き戻す
> 3. 移行完了後、`YUUKA_ENCRYPTION_SECRET_NEW` を `YUUKA_ENCRYPTION_SECRET` に昇格し、`YUUKA_ENCRYPTION_SECRET_NEW` を削除する
> 4. ローテーションのタイミングは運用判断とする（漏洩疑い時は即時、定期ローテーションは年1回を推奨）

#### 6.2.2 転送時暗号化（Encryption in Transit）

- 全ての通信はTLS 1.2以上を使用する
- Discord DMでの認証情報送受信は最小限に留め、可能な限りWeb管理UI（HTTPS）経由で行う

### 6.3 アクセス制御

#### 6.3.1 ユーザーレベルのアクセス制御

- パスワードマネージャのエントリは作成したユーザーのみがアクセス可能
- 他ユーザーへの共有機能は**提供しない**（ブラウザ操作の性質上、セキュリティリスクが高いため）
- エントリの作成・読取・更新・削除はすべてDiscordユーザーIDで認可チェックを行う

#### 6.3.2 LLMへのアクセス制御

- LLMはパスワードマネージャにFunction Call経由でのみアクセスできる
- LLMが取得できる情報は**必要最小限**とする
  - ブラウザ入力時: 該当サービスのユーザー名・パスワードのみ（他エントリは参照不可）
  - LLMのプロンプトや応答テキストに認証情報が含まれることを禁止する（ロガーでフィルタリング）
- パスワードマネージャへのアクセスはユーザーの明示的な指示があった場合のみ許可する

#### 6.3.3 操作ログ

- パスワードマネージャへのアクセス（読取・書込・削除）は全て監査ログに記録する
- ログに含まれる情報: タイムスタンプ・DiscordユーザーID・操作種別・対象サービス名（パスワード本体は記録しない）

### 6.4 LLMからの利用制限

- LLMによるパスワードマネージャ操作は以下のFunction Callのみに制限する

| Function名 | 説明 | 備考 |
|------------|------|------|
| `get_credential` | 指定サービスの認証情報を取得しブラウザに入力する | 返却値はブラウザコントローラに直接渡し、LLMの応答には含めない |
| `list_services` | 登録済みサービス名の一覧を返す | パスワードは含まない |
| `add_credential` | 新しい認証情報を登録する | ユーザー確認フロー必須 |
| `update_credential` | 既存認証情報を更新する | ユーザー確認フロー必須。新旧パスワードはLLM応答に含めない |
| `delete_credential` | 認証情報を削除する | ユーザー確認フロー必須 |

## 7. 会話履歴の永続化

### 7.1 保存方針

- 全ての送受信メッセージは **送信と同時にSQLiteへ永続保存**する（件数・期間の制限なし）
- スコープは `(user_id, bot_id, guild_id)`。`guild_id=NULL` は DM/秘書利用、非 NULL は汎用モードのギルド会話を表す
- Redis キャッシュは LLM への高速受け渡し用であり、正の履歴は SQLite が保持する。キャッシュキーは利用形態で分かれる: 秘書 `context:{botId}:secretary:{userId}`（直近15件）/ 汎用ギルド `context:{botId}:{guildId}`（直近30件）/ 汎用 owner DM `context:{botId}:dm:{ownerId}`（直近15件）
- Redis障害・再起動時はSQLiteから最新分を読み込んでキャッシュを再構築する
- **会話履歴クリア**: オーナーは統合管理画面から会話履歴をクリアできる。リセット境界 `context_floor:{botId}:{userId}` を記録し、それ以前のログを文脈復元から除外する（SQLite のログ自体は監査のため保持）

### 7.2 会話履歴データモデル（概略）

```
MessageLog {
  id              : UUID
  discord_user_id : DiscordUserId
  bot_id          : string         (どのBotとの会話か。分離キー)
  guild_id        : string (optional, NULL=DM/秘書 / 非NULL=汎用モードのギルド会話)
  discord_msg_id  : string         (DiscordメッセージID)
  role            : "user" | "assistant"
  content         : string
  reply_to_msg_id : string (optional, 返信元DiscordメッセージID)
  created_at      : datetime
}
```

### 7.3 返信チェーン解決の実装方針

- 返信メッセージを受信した際、`reply_to_msg_id` を辿ってSQLiteから再帰的に親メッセージを取得する
- ルートメッセージ（`reply_to_msg_id = NULL`）または設定した上限深度に達した時点で遡りを停止する
- 取得した返信チェーンは通常の15件キャッシュの**先頭に挿入**してコンテキストを構築する

## 8. バックアップ

### 8.1 方針

- SQLiteファイルをユーザー個人のGoogle Driveへバックアップする
- バックアップはユーザー単位でプライベートに保管される（他ユーザーのGoogle Driveには保存しない）
- バックアップ間隔はユーザーが管理UIまたはコマンドで自由に設定できる

### 8.2 バックアップ設定

| 項目 | 仕様 |
|------|------|
| バックアップ先 | ユーザー個人のGoogle Drive（専用フォルダ: `DiscordBot_Backup/`） |
| Google Drive認証 | OAuth 2.0（ユーザー毎に個別取得、スコープ: `https://www.googleapis.com/auth/drive.file`） |
| バックアップ間隔 | ユーザーが指定（最短: 1時間、最長: 30日） |
| ファイル形式 | SQLiteファイル（`.db`）をそのままアップロード、またはgzip圧縮 |
| 世代管理 | 直近N世代を保持（Nはユーザーが設定、デフォルト: 7） |
| 手動バックアップ | `Admin` ロールはコマンドで即時バックアップを実行できる |

### 8.3 リストア

- リストアは管理UIまたは手動操作で行う（自動リストアは提供しない）
- リストア手順は運用手順書で定義する

## 10. 非機能要件

| 項目 | 要件 |
|------|------|
| 可用性 | シングルインスタンス構成を前提とする。計画外停止時はcronジョブが再起動後に未送信リマインドを再処理する |
| レスポンスタイム | Discordへの返信: 通常10秒以内（LLM応答時間を除く） |
| セキュリティ | 保存データの暗号化（AES-256-GCM）、ユーザーデータの完全分離、TLS通信、監査ログ。HTTPS デプロイ時は **HSTS**（`Strict-Transport-Security: max-age=63072000; includeSubDomains`）を全レスポンスに付与し SSL ストリッピングを防ぐ。認証はタイミングオラクル対策（固定時間比較）でアカウント列挙を防止する |
| スケーラビリティ | 初期はシングルプロセス・SQLite構成とする。将来的なスケールアウトはDB移行（PostgreSQL等）で対応 |
| ログ・監査 | パスワードマネージャアクセス・ユーザー認証・管理操作イベントのログ記録 |
| バックアップ | ユーザー個人のGoogle DriveへSQLiteファイルをバックアップ（§8参照） |
| 実行環境 | Node.js（cronはNode.js上で動作するcronライブラリを使用） |
