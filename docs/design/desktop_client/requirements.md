# Yuuka Desktop — 要件定義

対象: Windows デスクトップ向け常駐オーバーレイ型チャットクライアント（Rust / egui）。
位置づけ: [index.md](index.md) の確定事項を前提とする要件の詳細化。提案段階（未実装）。

---

## 1. 目的と背景

- **目的**: Discord を開かずとも、デスクトップから常時 Yuuka に話しかけられるようにする。Discord と**同じ会話体験**（画像 / 音声 / テキスト入力 → 返答、タスク・予定・家計などのツール実行）を、軽量な常駐アプリで提供する。
- **背景**: 現状 Yuuka の対話入口は Discord のみ。PC 作業中に「ちょっと頼む」用途には Discord クライアントを開く手間がある。OS レベルのオーバーレイで即時呼び出せると価値が高い。
- **位置づけ**: 「新クライアント開発」イニシアチブの第一弾。将来 mac/Linux・モバイル等の他クライアントへ展開するため、**バックエンド API はクライアント非依存に設計**する（Discord にも特定 OS にも縛られない汎用チャット API）。

---

## 2. スコープ

### 2.1 In Scope（v1 で実装）

- トレイ常駐 + 画面上オーバーレイ（オーブ）。クリック / グローバルホットキーでチャットモーダルを開閉。
- **テキスト入力** → 返答（Markdown 表示）。
- **画像入力**（ファイル添付 / クリップボード貼り付け / ドラッグ&ドロップ）→ 返答。
- **音声入力**（マイク録音 → 送信、Gemini が文字起こし＋応答）→ **テキスト**返答。
- 返答の**ストリーミング表示**（少なくとも「考え中 / 入力中」ステータスと、重い処理の非同期完了プッシュ。トークン逐次は [architecture.md §6](architecture.md) の段階方針に従う）。
- 返答に含まれる**リッチコンテンツ**（Embed カード・グラフ PNG）の最低限の表示。
- **会話相手 Bot の選択**（ユーザーが所有/共有する Bot から選ぶ。既定は主 Bot）。Discord の同 Bot DM と会話コンテキストを共有。
- **ブラウザ OAuth デバイスフロー**でのログイン。トークンは Windows 資格情報マネージャに安全保存。
- **公開 URL を焼き込んだ本番ビルド**と、Web ダッシュボードからのダウンロード。

### 2.2 Out of Scope（v1 では作らない）

- **管理機能全般**（タスク/予定/家計/リマインド/ペルソナ/MCP/Bot 設定/PW マネージャ/Webhook/レポート 等）。これらは **Web ダッシュボードに集約**する（変更なし）。
- **音声出力（TTS）**。バックエンドに TTS が無いため将来フェーズ。
- **mac / Linux 版**。設計は移植可能にするが、ビルド/配布は v1 では Windows のみ。
- アプリ内でのユーザー登録・パスワード変更・Gemini キー設定。すべて Web で行う。
- 複数 Bot の同時並行チャット（タブ/分割）。v1 は 1 度に 1 つの会話。

### 2.3 非ゴール（明確に「やらない」）

- デスクトップにビジネスロジック（DB アクセス・暗号・Function 実装）を持たせない。**デスクトップは薄いプレゼンテーション層**であり、思考・記憶・ツール実行はすべてバックエンドが担う。
- ローカルに会話履歴やトークン以外の機微データを永続保存しない（履歴の正はサーバの `message_logs`）。

## 3. 想定ユーザーとシナリオ

- **想定ユーザー**: 既に Web で登録済み（Discord ID + パスワード + Gemini キー保有）の Yuuka 利用者。PC 常用者。
- **代表シナリオ**:
  1. PC 作業中、ホットキー（例: `Ctrl+Space` 二度押し / `Alt+Y`）でオーバーレイを呼び出し「明日 10 時に歯医者でリマインド」と打つ → 即座にリマインド登録の返答。
  2. スクショを撮ってオーバーレイに貼り付け「これ何のエラー？」→ 画像解析の返答。
  3. マイクボタンを押して話す「会議の議事を ToDo にして」→ 文字起こし＋ToDo 化提案。
  4. 重い処理（ブラウザ操作・検索クロール）を依頼 → 「やっておきます」の一時応答 → 完了時にオーバーレイへ完了プッシュ通知。

## 4. 機能要件（FR）

| ID | 要件 |
|---|---|
| FR-1 | トレイアイコン常駐。右クリックメニューで「表示 / 設定 / ログアウト / 終了」。 |
| FR-2 | 画面端にオーバーレイ（小さなオーブ）。**オーブ＝接続中 Bot のアイコン**（`discord_avatar_url`）のみを表示。透明背景・最前面・他アプリ操作を妨げない（オーブ以外はクリック透過）。位置はドラッグで移動・記憶。 |
| FR-2b | オーバーレイに**通知バッジ**: 未読返答 / 重い処理の完了プッシュがある場合、オーブの**右上に件数（数字）**を表示。モーダルを開くとクリアされる。 |
| FR-3 | オーブのクリックでチャットモーダルを開閉。フォーカス喪失 / `Esc` で閉じる（設定で「ピン留め」可）。**グローバルホットキーはプライマリ Bot のインスタンスのみが登録**（サブは登録せず衝突回避）。ホットキー→プライマリのモーダル表示。 |
| FR-4 | テキスト入力欄 + 送信（`Enter` 送信 / `Shift+Enter` 改行）。 |
| FR-5 | 画像添付: ① ファイル選択 ② クリップボード貼り付け（`Ctrl+V`）③ ドラッグ&ドロップ。送信前にサムネイル確認・取り消し可。 |
| FR-6 | 音声入力: マイクボタンで録音開始/停止。録音中は波形/インジケータ表示。停止で自動送信（または確認後送信、設定）。 |
| FR-7 | 返答表示: Markdown レンダリング（コードブロック・リスト・リンク）。リンクは既定ブラウザで開く。 |
| FR-8 | ステータス表示: 「考え中…」「入力中…」「（ツール名）実行中…」をストリームで反映。 |
| FR-9 | 重い処理の非同期完了をプッシュ受信し、モーダルが閉じていてもトレイ/オーバーレイで通知。 |
| FR-10 | リッチ返答: Embed カード（タイトル/本文/色/フィールド）とグラフ PNG をインライン表示。`richReplyEnabled=false` のユーザーはテキストのみ。 |
| FR-11 | 会話相手 Bot の選択（`GET /api/bots` から取得、既定は主 Bot）。**1 プロセス＝1 Bot**で、WS は選択中 Bot のみに接続。**モーダル表示中に Bot 切替可**＝切替時に現 WS を閉じ新 Bot で再接続（オーブのアイコンも切替先に変わる）。選択は記憶。 |
| FR-11b | **複数起動を許可**（Bot ごとに別プロセス＝別オーバーレイ）。同一 Bot の二重起動は抑止（既存インスタンスへフォーカス）。異なる Bot なら同時常駐可。 |
| FR-12 | 会話のクリア（コンテキストリセット）操作。サーバの会話リセット境界（`context_floor`）に合わせる。 |
| FR-13 | ログイン: ブラウザ OAuth デバイスフロー。未ログイン時はモーダルに「ブラウザでログイン」を提示。 |
| FR-14 | 設定: ホットキー、オーバーレイ位置/不透明度、起動時自動実行、相手 Bot、録音自動送信の ON/OFF。最小限。 |

## 5. 非機能要件（NFR）

| ID | 要件 | 目安 |
|---|---|---|
| NFR-1 軽量（最優先） | アイドル時のメモリ常駐を最小化。**複数起動（1 プロセス 1 Bot）で N 倍になるため、1 インスタンス当たりの常駐量が特に重要**。 | **1 インスタンス**の常駐 RAM 目標 **≤ 60MB**（オーバーレイのみ）、配布バイナリ **≤ 20MB**（単一 exe 推奨） |
| NFR-2 起動速度 | トレイ常駐からモーダル表示まで体感即時。 | ホットキー → 表示 **< 100ms** |
| NFR-3 省電力 | アイドル時は再描画を止める（イベント駆動）。 | アイドル CPU ≈ 0% |
| NFR-4 セキュリティ | トークンは OS 資格情報ストアに保存。平文ファイル保存禁止。通信は TLS（`wss`/`https`）。 | 端末ごとにトークン失効可（Web から管理） |
| NFR-5 データ分離 | 既存の不変条件を跨プロセスで維持。全 API はトークン由来の `user_id` でスコープ。 | [architecture/architecture_v2.md §0](../../architecture/architecture_v2.md) 継承 |
| NFR-6 障害分離 | バックエンド不通/WS 切断時はオフライン表示で穏当にデグレード。自動再接続（指数バックオフ）。 | 切断中の入力はローカル退避し再送 |
| NFR-7 移植性 | OS 依存（資格情報ストア・オーバーレイ・トレイ・ホットキー）を薄い抽象に隔離し、将来 mac/Linux へ移植可能に。 | — |
| NFR-8 配布 | 公開 URL をビルド時に焼き込み。Web からダウンロード。署名（将来）。 | — |

## 6. 制約・前提

- バックエンドは既存モノリス（`node:http`、本番 `https://yuuka.kawaii-music.moe` / コンテナ内 PORT 7854、リバースプロキシ TLS 終端）。WS は同一ポートの `upgrade` で受ける（プロキシの WebSocket 透過設定が必要）。
- 会話は**秘書モード相当**（個人 DM 相当）。Gemini キーは**ユーザー自身のキー**（`getUserGenAI(userId)`）を使用。→ 利用者は Web で Gemini キー設定済みであること。
- 既存の「新規 npm 依存は原則禁止」方針（[architecture_v2 §0.6](../../architecture/architecture_v2.md)）に対し、**WebSocket 実装の `ws` 追加は意図的な例外として承認済み**（2026-06-23）。実装時に §0.6 の依存リストへ追記する（[backend_api.md §2](backend_api.md) で詳述）。
- 現スキーマは **v12**。トークン管理テーブル追加は **v13** マイグレーションで冪等追加（既存テーブル不変）。
- **サーバー管理者が設定**（クライアントへ焼き込まない・既定値あり）: 添付上限 `DESKTOP_MAX_UPLOAD_MB`（既定 20MB）、デスクトップトークン TTL `DESKTOP_TOKEN_TTL_DAYS`（既定 90 日）。クライアントは `ready` の `maxUploadMb` や 401 応答に従う。詳細 [backend_api.md §8](backend_api.md)。
- **TTS（音声出力）は v1 非対応**。別途予定の音声クライアント開発が落ち着き次第、バックエンド TTS 増設のうえ追従する。

## 7. 受け入れ基準（v1 完了の定義）

1. Web からダウンロードした exe を起動 → トレイ常駐 → ブラウザ OAuth でログイン完了。
2. オーバーレイをホットキーで呼び出し、テキストで話しかけて返答が**ストリーム表示**される。
3. 画像（貼り付け）と音声（録音）でそれぞれ送信し、適切な返答が返る。
4. リマインド等の**ツール実行**が Discord 経由と同等に動作し、結果が会話に反映される。
5. 重い処理依頼で一時応答 → 完了プッシュがオーバーレイに届く。
6. 同じ Bot に Discord で DM した会話と**コンテキストが連続**している（記憶共有）。
7. Web ダッシュボードから当該端末のトークンを失効でき、失効後アプリが再ログインを促す。
8. アイドル時の常駐 RAM / CPU が NFR-1・NFR-3 の目安内。
